CORE

1. Classes & Objects — Data + Behavior, Together

▾

In C, structs hold data and functions operate on them — but they're completely separate. Nothing stops you from passing garbage to a function or forgetting to initialize a struct. The data and its operations are strangers.

C++ introduces classes: a blueprint that bundles data (members) and the functions that work on that data (methods) together. An object is an instance of a class. Now a BankAccount knows how to deposit — the logic lives with the data.

C++
class Car {
public:
    string brand;
    int speed;

    void drive() {
        cout << brand << " is driving at " << speed << " km/h\n";
    }
};

int main() {
    Car c;              // object created
    c.brand = "BMW";
    c.speed = 120;
    c.drive();         // BMW is driving at 120 km/h
}
💡 The mental shift: Stop thinking "data + functions" and start thinking "things that know how to do stuff". A Dog knows how to bark(). This is how humans already think about the world.

2. Access Specifiers — Who Can Touch What?

▾

Once data and functions live together, anyone can still reach in and corrupt the data: account.balance = -99999; — nothing stops this. The class has rules (invariants) that can be broken from outside.

Access specifiers let the class control who sees what. private members are only accessible inside the class itself. public members form the safe, validated interface. The class guarantees its invariants are maintained.

SpecifierInside ClassDerived ClassOutside
public✓✓✓
protected✓✓✗
private✓✗✗
🔑 The real reason: Access control isn't about secrecy — it's about protecting invariants. A class has rules (e.g., balance ≥ 0). Making data private ensures those rules can't be broken by external code. You're protecting correctness, not hiding secrets.

3. Constructors & Destructors — Automatic Lifecycle Management

▾

In C, you create a struct, then separately call an init function. Forget to init → garbage data. Forget to free → memory leak. The creation and cleanup are manual, error-prone, and disconnected from the object's lifetime.

Constructors run automatically when an object is created. Destructors run automatically when it's destroyed. You literally cannot forget — the language guarantees it. This pattern is called RAII (Resource Acquisition Is Initialization).

C++
class File {
    FILE* handle;
public:
    File(const char* name) {   // Constructor — auto-called on creation
        handle = fopen(name, "r");
    }
    ~File() {                      // Destructor — auto-called on destruction
        fclose(handle);
    }
};
// Usage: File opens automatically, closes when it goes out of scope
{
    File f("data.txt");   // opened
    // use f...
}                              // closed automatically — CANNOT forget
💡 RAII is C++'s superpower: The constructor acquires resources. The destructor releases them. The scope of the object controls the lifetime of the resource. No garbage collector needed — deterministic, predictable, automatic.

4. Encapsulation — The Black Box Philosophy

▾

Encapsulation isn't a keyword — it's the philosophy that emerges from private data + public methods. The class becomes a black box: interact through buttons (public methods), never reach inside and turn gears directly.

Without encapsulation: Any code anywhere can modify balance. If you later change how balance is stored internally (say from double to int cents), every piece of code that touched balance directly breaks.

With encapsulation: Only the class touches balance. External code uses deposit() and getBalance(). Change the internal representation freely — nothing outside breaks because nothing outside directly touched it.

🎯 The design goal: Large private surface, small public surface. The more you expose, the more you're locked into. The less you expose, the more freedom you have to change your implementation without breaking anything.

5. Inheritance — Reuse & Extend Through "IS-A"

▾

You have Car and Truck. Both have brand, speed, start(), stop(). You're duplicating code. Worse — fix a bug in Car::start() and you must also fix it in Truck::start(). This doesn't scale.

Extract common parts into a base class (Vehicle). Car and Truck inherit everything and only define what's unique. One place to fix bugs. Clean "IS-A" relationships.

C++
class Vehicle {          // Base class
public:
    string brand;
    void start() { cout << "Starting...\n"; }
};

class Car : public Vehicle {   // Car IS-A Vehicle
public:
    int numDoors;           // only Cars have doors
};

class Truck : public Vehicle { // Truck IS-A Vehicle
public:
    double cargoCapacity;   // only Trucks have this
};
⚠️ Inheritance types matter: public inheritance = "IS-A" (most common). protected/private inheritance = "implemented-in-terms-of" (rare — prefer composition instead).

6. Polymorphism — One Interface, Many Behaviors

▾

You have a collection of shapes. You want to call area() on each. But at compile time, you don't know which shape you're dealing with. Without polymorphism, the compiler always calls Shape::area() — wrong for every actual shape.

Mark the function virtual. The compiler adds a hidden vptr (virtual pointer) to every object, pointing to a vtable of the object's actual functions. At runtime, the correct function is dispatched — dynamic dispatch.

C++
class Shape {
public:
    virtual double area() { return 0; }  // virtual = can be overridden
    virtual ~Shape() {}                   // virtual destructor — MANDATORY
};

class Circle : public Shape {
    double r;
public:
    double area() override { return 3.14159 * r * r; }
};

class Rectangle : public Shape {
    double w, h;
public:
    double area() override { return w * h; }
};

// Runtime dispatch — the magic:
Shape* shapes[] = { new Circle(5), new Rectangle(4, 6) };
for (auto s : shapes)
    cout << s->area() << "\n";  // calls the RIGHT area() at runtime!
🧠 How it works internally: Every object with virtual functions gets a hidden vptr (8 bytes on 64-bit). The vptr points to the class's vtable — an array of function pointers. When you call s->area(), the runtime follows vptr → vtable → actual function. One extra pointer dereference per call — negligible cost for enormous design flexibility.

7. Abstract Classes & Interfaces — Enforce a Contract

▾

Shape shouldn't define area() at all — a generic "shape" has no area formula. But you want to enforce that every derived class provides one. How do you say "you MUST implement this"?

Use pure virtual functions (= 0). The class becomes abstract — uninstantiable. Every concrete subclass must override every pure virtual function. This is C++'s way of doing interfaces.

C++
class Shape {                    // Abstract — cannot instantiate
public:
    virtual double area() = 0;         // pure virtual = "you MUST implement this"
    virtual void draw() = 0;
    virtual ~Shape() {}             // virtual destructor
};

// Shape s;  ← ERROR: abstract class

class Circle : public Shape {
    double r;
public:
    double area() override { return 3.14 * r * r; }  // MUST exist
    void draw() override { /* render */ }          // MUST exist
};
💡 Interfaces in C++: A class with only pure virtual functions is a pure interface. It defines a contract that implementors must fulfill. Write code that depends on the contract, not on any specific implementation.

8. Static Members — Belongs to the Class, Not Objects

▾

Some data genuinely belongs to the class, not to any individual object. How many Counter objects exist right now? That's not per-object data — it's class-wide data. Where do you put it?

static members have one copy shared by all instances. They exist even before any object is created. They live in static storage — not inside any object's memory layout.

C++
class Counter {
    static int total;        // ONE copy, shared by ALL objects
    int id;
public:
    Counter()  { total++; id = total; }
    ~Counter() { total--; }
    static int getTotal() { return total; }  // no 'this' pointer
};
int Counter::total = 0;   // defined outside — lives in static storage

Counter a, b, c;
cout << Counter::getTotal();  // 3 — called on the CLASS, not an object
sizeof(Counter);                // only counts 'id' — total is not inside objects

9. Const & Mutable — Promise Not to Change

▾

const means "I promise this won't change." The compiler enforces this promise. It appears in more positions than any other qualifier — variable, pointer, parameter, method, return type.

C++
// Four pointer-const combinations (read right-to-left):
int* p1 = &x;            // pointer to int — both changeable
const int* p2 = &x;       // pointer to const int — value locked, pointer free
int* const p3 = &x;       // const pointer to int — pointer locked, value free
const int* const p4 = &x;  // const pointer to const int — both locked

// Const member function — cannot modify the object:
class Circle {
    double radius;
public:
    double getRadius() const { return radius; }  // const method — 'this' is const
    void setRadius(double r) { radius = r; }       // non-const — can modify
};
const Circle c(5.0);
c.getRadius();  // OK
// c.setRadius(3);  ← ERROR: non-const method on const object

// mutable — escape hatch from const:
class Cache {
    mutable bool dirty = true;   // can change even in const methods
    mutable double cachedVal;
public:
    double getValue() const {      // logically const
        if (dirty) { cachedVal = compute(); dirty = false; }
        return cachedVal;
    }
};

10. Friend — Controlled Access Exception

▾

Sometimes a non-member function genuinely needs access to private internals — like operator<< for stream output. It can't be a member (the left side is cout, not your class), but it needs to read private data.

friend grants a specific external entity (function, method, or entire class) full access to all private and protected members. The class itself declares who it trusts — friendship is granted, not taken.

C++
class Point {
    double x, y;
public:
    // operator<< is a FREE function but needs private access:
    friend ostream& operator<<(ostream& os, const Point& p) {
        return os << "(" << p.x << ", " << p.y << ")";
    }
};
Point p{3.0, 4.0};
cout << p;  // prints: (3.0, 4.0)

// Friend class — entire class gets access:
class TreeNode {
    int value;
    TreeNode *left, *right;
    friend class BinaryTree;  // BinaryTree can touch everything
};
⚠️ Friendship rules: Not inherited. Not transitive. Not symmetric. Use sparingly — it breaks encapsulation intentionally. Prefer making things public over friending when possible.
DEEP DIVE

11. Virtual Functions — The VTable Mechanism

▾

This is the crown jewel of OOP. Understanding how virtual actually works makes everything click.

🧠 Memory layout of a polymorphic object:

[ vptr ] → Dog's vtable: [ speak → Dog::speak, eat → Animal::eat, ... ]
[ name ] (inherited from Animal)
[ breed ] (Dog's own members)

Every class with virtual functions gets a vtable. Every object gets a vptr (8 bytes on 64-bit). When you call a virtual function, the runtime follows vptr → vtable → actual function pointer. One indirection = negligible cost.
⚠️ Why virtual destructor is MANDATORY: If you delete a Shape* that actually points to a Circle, and the destructor isn't virtual, only Shape::~Shape() runs. Circle's destructor never runs → memory leak. virtual ~Shape() ensures the correct destructor chain through the vtable.

12. Operator Overloading — Natural Syntax for Your Types

▾

User-defined types feel like second-class citizens: Vector v = add(v1, v2); vs int a = 1 + 2;. Why can't your types use natural operator syntax?

C++ lets you define what +, ==, [], (), etc. mean for your types. You're not changing operators — you're teaching them a new vocabulary. The operator's concept stays the same; only the implementation changes.

C++
class Vector {
    double x, y;
public:
    Vector operator+(const Vector& o) const { return {x+o.x, y+o.y}; }
    Vector& operator+=(const Vector& o) { x+=o.x; y+=o.y; return *this; }
    bool operator==(const Vector& o) const { return x==o.x && y==o.y; }
    double& operator[](int i) { return i==0 ? x : y; }
    Vector operator-() const { return {-x, -y}; }  // unary minus
};

Vector v = v1 + v2;     // natural!
v += v3;                // natural!
if (v == v4) { }        // natural!
cout << v[0];          // array-like access

13. Class Lifecycle — What Executes When

▾
⏱️ Complete execution order for an object:

1. Memory allocated (stack shift or heap malloc)
2. Base class constructors run (inheritance declaration order)
3. Member initializer list runs — in DECLARATION order (not list order!)
4. Constructor body {} executes
5. Object is fully alive ✓
6. ... object is used ...
7. Destructor body {} executes
8. Members destroyed in REVERSE declaration order
9. Base destructors run in REVERSE inheritance order
10. Memory freed
💡 Why the initializer list uses declaration order: Destruction must mirror construction. If construction order depended on the initializer list (which could vary per constructor), the destructor wouldn't know the order without storing it at runtime. By tying both to declaration order, the compiler guarantees they mirror without any runtime tracking.
⚠️ Priority for member initialization:
1. Constructor initializer list (highest priority)
2. In-class member initializer (fallback, C++11+)
3. Default initialization (for primitives: garbage; for class types: default constructor)
HISTORY

14. C++ Version Evolution — Why Each Feature Was Added

▾
C++98/03

The Foundation — "How do we model real-world entities?"

Classes, inheritance, virtual functions, polymorphism, templates, RTTI, operator overloading, namespaces. Everything you think of as "classic C++ OOP" lives here.

C++11

The Revolution — "How do we make this safe, fast, and less painful?"

override/final, move semantics, smart pointers, auto, lambdas, constexpr, nullptr, delegating constructors, uniform initialization, noexcept, = default/= delete. The biggest modernization in C++ history.

C++14

The Polish — "Fix the rough edges we missed"

Generic lambdas, relaxed constexpr, return type deduction, [[deprecated]].

C++17

Structural Completeness — "Fill gaps in the type system"

Structured bindings, if constexpr, std::optional, std::variant, class template argument deduction (CTAD), [[nodiscard]], inline variables.

C++20

The Second Revolution — "Constraints, async, and faster builds"

Concepts, coroutines, modules, ranges, spaceship operator <=>, consteval, constinit.
